# 데이터 구축 가이드: AI_SYS

작성일: 2026-04-08  
문서 버전: v1.0

> 안내: 본 문서는 AI_SYS의 판례 데이터 구축과 내부 DB 적재를 위한 실행 가이드다. 향후 데이터 소스 정책, 구현 범위, 운영 결과에 따라 내용이 변경될 수 있다.

## 1. 문서 목적
이 문서는 AI_SYS에서 사용할 판례 원문과 메타데이터를 어떤 방식으로 확보, 정제, 적재, 검수, 운영할지 상세 절차를 정의한다.  
핵심 목표는 외부 API 의존도를 낮추고, 내부 DB 기반 검색/요약 품질을 안정적으로 확보하는 것이다.

## 2. 구축 범위
- 대상 데이터: 경찰 시험 대비에 필요한 판례 데이터
- 우선 과목: 형법, 형사소송법, 경찰학
- 우선 단위: 사건번호가 명확한 빈출 판례
- 운영 기준: 검수 완료 데이터만 사용자 응답에 사용

## 3. 데이터 구축 원칙
- 기준 데이터 우선: 서비스 응답의 기준은 내부 DB 적재 데이터로 고정한다.
- 원문 보존: 판례 원문은 가능한 한 원본에 가깝게 보관한다.
- 메타데이터 구조화: 검색, 필터링, 복습에 필요한 필드를 분리 저장한다.
- 단계별 검수: 수집, 정규화, 요약, 공개 단계마다 품질 점검을 수행한다.
- 추적 가능성: 모든 데이터에 출처, 수집일, 수정일, 상태를 기록한다.

## 4. 소스 확보 전략
### 4.1 우선 소스
- 공식 공개 파일데이터
- 공식 사이트에서 열람 가능한 판례 출처
- 운영자 수집/정리 자료

### 4.2 소스 사용 정책
- 이용약관, 재사용 조건, 저작권 안내를 먼저 확인한다.
- 자동 수집 필요 시 허용 범위 확인 후 진행한다.
- 허용 범위가 불명확한 데이터는 임시 보류 상태로 저장한다.

### 4.3 수집 단위
- 1차: 경찰 시험 빈출 판례 300~1000건
- 2차: 오답/검색 로그 기반 확장 판례

## 5. 내부 DB 데이터 모델
### 5.1 핵심 테이블
- cases: 판례 기본 정보 + 원문
- case_chunks: 검색/RAG용 청크
- case_keywords: 검색 키워드
- case_relations: 유사/관련/반대 취지 판례 연결
- user_case_history: 사용자 조회/오답/복습 이력

### 5.2 최소 메타데이터 필드
- 사건번호
- 판례명
- 법원명
- 선고일
- 과목
- 핵심 쟁점
- 판결 요지
- 시험 포인트
- 출처 링크
- 수집일
- 수정일
- 데이터 상태

### 5.3 데이터 상태값 권장안
- collected: 수집 완료
- parsed: 정규화 완료
- summarized: 요약 생성 완료
- reviewed: 검수 완료
- published: 서비스 공개 가능
- blocked: 사용 보류

## 6. 적재 파이프라인 상세
### 단계 1. 원문 확보
- 원문 텍스트와 기본 출처 정보를 수집한다.
- 사건번호가 없거나 출처가 불명확하면 임시 보류한다.

### 단계 2. 기본 메타데이터 추출
- 사건번호, 판례명, 법원명, 선고일, 출처 링크를 추출한다.
- 형식 흔들림이 있는 필드는 정규화 규칙으로 통일한다.

### 단계 3. 기본 적재
- cases에 원문과 기본 메타데이터를 저장한다.
- 상태값은 collected로 시작한다.

### 단계 4. 정규화 및 중복 제거
- 사건번호 포맷 통일
- 법원명 표준화
- 과목 태깅
- 중복 판례 제거
- 누락 필드 점검

### 단계 5. 청크 분할
- 원문을 검색과 RAG에 맞는 단위로 분할한다.
- 권장: 문단 기준 또는 토큰 기준 고정 길이 분할
- 결과를 case_chunks에 저장한다.

### 단계 6. 키워드/요약 메타데이터 생성
- 핵심 쟁점
- 결론 요약
- 시험 포인트
- 검색 키워드
- 필요 시 LLM 초안 생성 후 검수 반영

### 단계 7. 임베딩 및 인덱싱
- 청크별 임베딩 생성
- 벡터 검색 인덱스 구축
- 사건번호 exact match 인덱스와 키워드 인덱스 병행

### 단계 8. 검수 및 공개
- reviewed 검수 기준 통과 시 published로 변경
- published 데이터만 검색/요약 응답에서 사용

## 7. 검색 입력별 처리 방식
### 7.1 사건번호 입력
- 내부 DB exact match 우선
- 미등록 시 키워드 검색 또는 등록 대기 안내

### 7.2 키워드 입력
- case_keywords + cases 메타데이터 검색
- 필요 시 임베딩 검색 + reranking 적용

### 7.3 OCR 입력
- OCR 텍스트에서 핵심 키워드 추출
- 내부 DB 검색으로 연결
- OCR은 입력 보조 수단이며 원문 확보 수단이 아니다

## 8. 품질 검수 기준
### 8.1 필수 검수 항목
- 사건번호 정확성
- 출처 링크 유효성
- 원문 누락 여부
- 과목 분류 정확성
- 쟁점/결론/시험 포인트 일관성

### 8.2 검색 품질 기준
- 사건번호 검색 성공률
- 상위 N개 결과 적합도
- 오답 연결 판례의 재검색 성공률

### 8.3 요약 품질 기준
- 쟁점 누락 여부
- 결론 왜곡 여부
- 출처 표시 여부
- 시험 포인트 실용성

## 9. 운영 절차
### 9.1 배치 주기 권장
- 일간: 신규 수집 후보 등록, 실패 로그 확인
- 주간: 정규화/중복 정리, 요약 검수
- 월간: 취약 쟁점 확장, 키워드 체계 보정

### 9.2 역할 분담 예시
- 데이터 담당: 수집, 정규화, 중복 정리
- 도메인 검수 담당: 법리/쟁점 검수
- 서비스 담당: 검색/요약 품질 지표 점검

## 10. 위험과 대응
- 소스 정책 불명확: 사용 보류 처리 후 공식 확인
- 사건번호 누락: 공개 제외, 보완 대기
- 요약 품질 저하: reviewed 이전 상태 유지
- 검색 실패 증가: 키워드 보강, 청크 전략 재조정

## 11. MVP 수용 기준
- 초기 목표 건수 판례 적재 완료
- 사건번호 검색이 안정적으로 동작
- 핵심 키워드 검색 품질 기준 통과
- published 상태 데이터만 서비스 응답에 사용
- 오답/복습 이력과 판례 데이터 연결 가능

## 12. 즉시 실행 체크리스트
- 데이터 소스 정책 확인
- 최소 메타데이터 스키마 확정
- 초기 판례 1차 목록 확정
- 수집/정규화/적재 스크립트 작성
- 검수 체크리스트 확정
- 검색/요약 샘플 테스트 실행

## 13. 원문 수집 작업표 템플릿
아래 템플릿은 운영자가 판례 원문 확보 상태를 추적할 때 사용하는 기준 표다.

| 수집ID | 우선순위 | 과목 | 사건번호 | 판례명 | 소스유형 | 출처URL | 원문확보 | 메타입력 | 정규화 | 검수상태 | 담당자 | 최종수정일 | 비고 |
|---|---|---|---|---|---|---|---|---|---|---|---|---|---|
| C-0001 | P1 | 형사소송법 | 2019도12345 | 위법수집증거 관련 판결 | 공식사이트 | https://example.org/case/2019do12345 | Y | Y | Y | reviewed | 홍길동 | 2026-04-08 | 유사판례 2건 연결 |
| C-0002 | P1 | 형법 | 2020도56789 | 정당방위 범위 판결 | 파일데이터 | https://example.org/case/2020do56789 | Y | N | N | collected | 김민수 | 2026-04-08 | 메타 보완 필요 |
| C-0003 | P2 | 경찰학 | 2018도11111 | 현행범 체포 적법성 판결 | 공식사이트 | https://example.org/case/2018do11111 | N | N | N | blocked | 이수진 | 2026-04-08 | 출처 이용조건 확인 필요 |

### 13.1 컬럼 정의
- 수집ID: 수집 작업의 고유 식별자
- 우선순위: P1, P2, P3로 구분되는 작업 긴급도
- 과목: 형법, 형사소송법, 경찰학 등 분류
- 사건번호: 내부 DB exact match의 핵심 키
- 판례명: 운영자가 판별 가능한 판례 제목
- 소스유형: 공식사이트, 파일데이터, 운영자자료 등
- 출처URL: 원문 근거 링크
- 원문확보: 원문 텍스트 확보 여부 (Y/N)
- 메타입력: 최소 메타데이터 입력 여부 (Y/N)
- 정규화: 포맷 통일과 중복 처리 완료 여부 (Y/N)
- 검수상태: collected, parsed, summarized, reviewed, published, blocked
- 담당자: 수집 또는 검수 담당자
- 최종수정일: 마지막 변경 날짜
- 비고: 이슈, 보완 요청, 유사 판례 연결 메모

### 13.2 권장 운영 규칙
- P1 판례는 원문확보와 메타입력을 동일 주차 내 완료한다.
- 검수상태가 reviewed 이상인 데이터만 published 후보로 승격한다.
- blocked 데이터는 비고에 차단 사유를 필수 기록한다.
- 사건번호 중복 발견 시 신규 행 생성 대신 기존 행을 갱신한다.

### 13.3 CSV 템플릿 예시

## 14. DB 구축 실행 가이드 (PostgreSQL)

### 14.1 적용 대상
- DB 엔진: PostgreSQL 15+
- 확장: pgcrypto, pgvector
- 스키마 파일: `db/schema.sql`

### 14.2 로컬 실행 순서
1. PostgreSQL을 설치하고 실행한다.
2. DB와 계정을 생성한다.
3. 스키마 파일을 적용한다.
4. 시드 데이터를 적재한다.

```bash
# 예시: DB 생성
createdb aisys

# 스키마 적용
psql -d aisys -f db/schema.sql

# 시드 데이터 적용
psql -d aisys -f db/seed.sql
```

### 14.3 생성되는 핵심 객체
- enum 타입
	- data_status: collected, parsed, summarized, reviewed, published, blocked
	- relation_type: similar, related, opposite
- 테이블
	- cases
	- case_chunks
	- case_keywords
	- case_relations
	- user_case_history
- 뷰
	- published_cases (서비스 공개 가능 데이터만 조회)
- 트리거
	- cases 업데이트 시 updated_at 자동 갱신

### 14.4 운영 규칙
- 신규 수집 데이터는 cases.status를 collected로 시작한다.
- 검수 통과 전에는 published 상태로 승격하지 않는다.
- 서비스 검색/요약 API는 published_cases 또는 status=published 조건만 사용한다.
- 사건번호는 cases.case_number unique 제약으로 중복 적재를 방지한다.

### 14.5 초기 점검 쿼리
```sql
-- 상태별 건수
SELECT status, COUNT(*)
FROM cases
GROUP BY status
ORDER BY status;

-- 공개 가능한 판례 샘플
SELECT case_number, case_name, subject, updated_at
FROM published_cases
ORDER BY updated_at DESC
LIMIT 20;
```

## 15. 수집 데이터 저장 위치 표준

수집한 판례 원문 및 정제 산출물은 아래 경로를 사용한다.

- `code/data/raw/`: 수집 원문(원본)
- `code/data/normalized/`: 정규화 완료 데이터
- `code/data/reviewed/`: 검수 완료 데이터
- `code/data/failed/`: 실패/반려 데이터
- `code/data/manifests/`: 수집/적재 이력 메타 파일

운영 원칙
- 원문(raw) 데이터는 Git 커밋 대상에서 제외하고 외부 저장소 보관을 우선한다.
- 게시 파이프라인 입력은 reviewed 데이터만 사용한다.
- 모든 배치 실행 결과는 manifests에 이력으로 남긴다.
