# PRD: AI_SYS

작성일: 2026-04-02  
문서 버전: v1.0

> 안내: 본 문서는 현재 기획 및 정리 기준으로 작성되었으며, 추후 개발 과정에서 요구사항, 구현 범위, 검증 결과에 따라 내용이 변경될 수 있습니다.

## 1. 제품 개요
AI_SYS는 경찰 공무원 시험 준비생이 판례를 더 빠르게 찾고, 더 정확하게 이해하고, 반복적으로 복습할 수 있도록 돕는 AI 기반 학습 지원 시스템이다.  
핵심 가치는 판례 검색, 근거 기반 요약, 문제 풀이 연계, 오답 기반 복습을 하나의 학습 흐름으로 묶어 수험 효율을 높이는 데 있다.

## 2. 문제 정의
AI_SYS가 해결하려는 핵심 문제는 수험생이 판례를 찾고 검증하는 데 많은 시간이 들고, 일반 AI 도구만으로는 시험 준비에 필요한 정확도와 근거 신뢰성을 확보하기 어렵다는 점이다.

### 2.1 현황과 배경
- 경찰 공무원 시험 준비 과정에서는 판례 이해와 암기가 중요한 비중을 차지한다.
- 수험생은 문제를 풀다가 판례가 기억나지 않으면 검색 사이트, 교재, 요약 노트, 일반 AI 도구를 반복적으로 오가게 된다.
- 이 과정은 학습 흐름을 끊고, 핵심 쟁점보다 검색 자체에 시간을 소모하게 만든다.

### 2.2 사용자 문제 (Pain Points)
- 필요한 판례를 즉시 찾기 어렵다.
- 판례를 찾더라도 시험 관점에서 어떤 부분이 중요한지 다시 정리해야 한다.
- 일반 AI 도구는 출처가 불명확하거나 부정확한 답변을 제공할 수 있다.
- 오답으로 연결된 판례를 체계적으로 누적하고 복습하기 어렵다.

### 2.3 기존 대안의 한계
- 일반 검색 서비스는 원문 탐색에는 유용하지만 수험생 관점의 핵심 요약과 문제 연계가 약하다.
- 일반 LLM은 빠른 설명은 가능하지만 판례 출처와 최신성 보장이 부족할 수 있다.
- 수기 오답노트는 개인화에는 유리하지만 관리 비용이 크고 반복성이 떨어진다.

### 2.4 왜 지금 해결해야 하는가
- 수험생은 더 짧은 시간 안에 더 많은 판례를 이해하고 암기해야 한다.
- RAG 기반 검색과 요약 기술을 활용하면 근거 중심 응답 품질을 일정 수준까지 확보할 수 있다.
- 초기 단계부터 오답과 복습 루프를 설계하면 단순 검색 도구가 아니라 학습 성과를 높이는 시스템으로 발전시킬 수 있다.

## 3. 목표
- 판례를 키워드나 사건명으로 빠르게 찾을 수 있게 한다.
- 판례의 핵심 쟁점, 결론, 시험 포인트를 짧고 정확하게 제공한다.
- 관련 문제 풀이와 오답 복습까지 연결되는 학습 흐름을 제공한다.
- 가능한 범위에서 근거 출처가 포함된 응답을 제공한다.

## 4. 비목표 (Out of Scope)
- 법률 자문이나 실제 법적 판단 제공
- 모든 법 분야를 포괄하는 범용 법률 플랫폼 구축
- 초기 버전에서 완전한 자동 채점 시스템 구현
- 초기 버전에서 실시간 다중 사용자 협업 기능 제공

## 5. 타겟 사용자
- 주요 사용자: 경찰 공무원 시험 준비생
  - 주요 니즈 1: 문제 풀이 중 필요한 판례를 빠르게 찾고 싶다.
  - 주요 니즈 2: 판례의 핵심 쟁점을 시험 관점에서 정리된 형태로 보고 싶다.
  - 주요 니즈 3: 자주 틀리는 판례와 쟁점을 반복 복습하고 싶다.
- 확장 사용자: 경찰학과, 행정학과 등 판례 기반 학습이 필요한 학생

## 6. 핵심 사용자 시나리오
시나리오명: 기출 문제 풀이 중 필요한 판례를 찾아 이해하고 복습까지 이어가는 수험생

1. 문제 풀이 중 막힘
- 사용자는 형법 또는 형사소송법 문제를 풀다가 특정 판례가 기억나지 않는다.
- 시스템은 키워드 입력, 문제 지문 사진 촬영(OCR), 판례 번호 직접 입력 중 하나를 선택할 수 있는 검색 진입점을 제공한다.

2. 판례 검색
- 사용자는 문제 지문을 카메라로 촬영하거나, 해설지의 판례 번호를 입력하거나, 키워드를 직접 입력하여 검색을 시작한다.
- 사진 촬영 시 OCR로 텍스트가 추출되어 자동 검색이 시작된다.
- 판례 번호 입력 시 내부 DB에서 사건번호 기준으로 판례를 우선 조회하고, 관련 원문 청크를 기반으로 LLM 요약으로 연결된다.
- 시스템은 관련 판례 목록과 핵심 메타정보를 반환한다.

3. 판례 요약 확인
- 사용자는 판례 상세 결과를 열어 쟁점, 결론, 시험 포인트를 확인한다.
- 시스템은 출처와 함께 요약을 제공하고, 유사 판례 목록도 사건번호와 함께 제시한다.

4. 문제 적용
- 시스템은 해당 판례와 연결된 기출 또는 예상 문제를 제공한다.
- 사용자는 문제를 풀며 판례 이해도를 점검한다.

5. 오답 저장 및 복습
- 사용자가 문제를 틀리거나 헷갈리면 해당 판례를 오답으로 저장한다.
- 시스템은 취약 판례와 쟁점을 기반으로 복습 우선순위를 제안한다.

## 7. 기능 요구사항

### FR-1. 판례 검색
- 사용자는 사건명, 키워드, 쟁점 단위로 판례를 검색할 수 있어야 한다.
- 사용자는 문제 지문을 앱 내 카메라로 촬영하면 OCR을 통해 텍스트가 추출되고 자동으로 검색이 시작되어야 한다.
- 사용자는 해설지에 표기된 판례 번호를 직접 입력하여 해당 판례를 즉시 조회할 수 있어야 한다.
- 시스템은 검색 결과에 판례명, 관련 주제, 간단 요약을 표시해야 한다.

### FR-2. 판례 요약 및 출처 제공
- 시스템은 내부 DB에 적재된 판례 원문과 메타데이터를 기준 데이터로 사용해야 한다.
- 판례 번호 입력 시 내부 DB에서 사건번호 기준 조회를 우선 수행하고, 필요한 경우 공식적으로 이용 가능한 외부 출처 링크를 참조 정보로 제공해야 한다.
- 조회된 판례 원문 또는 관련 청크는 LLM에 입력되어 쟁점, 결론, 시험 포인트, 유사 판례 정보를 요약·출력해야 한다.
- 유사 판례는 요약, 판례 번호, 해당 판례로 이동할 수 있는 링크와 함께 제공해야 한다.
- 응답에는 가능한 범위에서 출처 또는 원문 참조 정보를 함께 표시해야 한다.

### FR-3. 질의응답 보조
- 사용자는 판례와 관련된 자연어 질문을 입력할 수 있어야 한다.
- 시스템은 검색된 판례 문맥을 기반으로 답변을 생성해야 한다.
- 문맥 일치도가 낮은 경우 단정형 응답보다 관련 후보 판례를 우선 제시해야 한다.

### FR-4. 문제 풀이 연계
- 시스템은 판례와 연결된 기출 문제 또는 예상 문제를 제공해야 한다.
- 사용자는 문제 풀이 결과를 확인하고 이해도를 점검할 수 있어야 한다.

### FR-5. 오답 및 복습 관리
- 사용자는 틀린 문제와 관련 판례를 오답 항목으로 저장할 수 있어야 한다.
- 시스템은 오답 이력과 취약 쟁점을 기반으로 복습 목록을 추천해야 한다.

### FR-6. 학습 대시보드 (홈 화면)
- 앱 홈 화면은 복습 대시보드로 구성된다.
- 시스템은 오늘 복습 추천 판례 목록, 최근 오답 기록, 판례 검색 진입 버튼을 홈에서 바로 제공해야 한다.
- 사용자는 재방문 시 별도 탐색 없이 홈에서 바로 복습을 시작할 수 있어야 한다.

### FR-7. 데이터 품질 관리
- 판례 데이터는 검색 가능한 구조로 정리되고 지속적으로 검수되어야 한다.
- 잘못된 매핑, 중복 데이터, 누락된 출처는 검수 대상로 분리 관리해야 한다.

## 8. 비기능 요구사항
- 성능: 일반 검색/요약 요청은 사용자가 대기 가능하다고 느끼는 수준으로 응답해야 한다.
- 신뢰성: 검색 실패, 출처 누락, 데이터 부족 상황을 사용자에게 명확히 알려야 한다.
- 사용성: 비전공자도 검색에서 복습까지의 흐름을 빠르게 이해할 수 있어야 한다.
- 보안: 민감정보나 비공개 데이터는 코드에 하드코딩하지 않는다.
- 확장성: 판례 과목 확대와 데이터 증가를 감당할 수 있는 구조를 지향한다.

## 9. 현재 기술/시스템 범위
- 현재 저장소 범위: 초기 기획 문서와 프로젝트 방향 정의
- 목표 서비스 범위:
  - Frontend: 검색, OCR 촬영, 요약, 오답/복습 인터페이스
  - Backend: 판례 검색, 요약, 질의응답 API
  - 데이터 저장: 내부 판례 원문/메타데이터 DB 구축, 검색용 청크 및 키워드 인덱스 관리
  - 외부 연동: 공식 공개 API 또는 이용 가능한 공식 출처 링크를 참조 경로로 검토
  - AI/데이터: RAG, 임베딩 검색, 출처 기반 응답 구조, OCR 텍스트 추출
  - 품질 관리: 사용자 시나리오 테스트, 검색 정확도 점검

## 10. 성공 지표 (KPI)
- 검색 성공률: 사용자가 원하는 판례를 몇 번의 시도 안에 찾는지 측정
- 요약 도달률: 검색 후 상세 요약 화면까지 도달하는 비율
- 문제 연계 사용률: 판례 조회 후 문제 풀이로 이어지는 비율
- 복습 재방문율: 오답 저장 후 다시 복습 화면을 사용하는 비율
- 응답 신뢰도: 사용자 피드백 기준으로 요약/답변 만족도를 측정

## 11. 마일스톤
- M1 (1~3주): 문제 정의 정교화, 사용자 인터뷰, 핵심 문서 세트 정리
- M2 (4~6주): 데이터 수집 범위 확정, 검색 구조 및 서비스 흐름 설계
- M3 (7~10주): 검색/요약 프로토타입 구현, 핵심 사용자 시나리오 검증
- M4 (11~13주): 문제 연계 및 오답 복습 기능 확장, UX 개선
- M5 (14~15주): 통합 테스트, 발표 시나리오 정리, 최종 시연 준비

## 12. 데이터 및 운영 가정

### 12.1 데이터 범위
- 초기에는 경찰 시험 준비에 직접적으로 필요한 판례 범위를 우선 수집한다.
- 형법, 형사소송법, 경찰학 관련 주요 판례를 우선순위 대상으로 둔다.

### 12.2 데이터 품질 이슈
- 판례명 표기 흔들림
- 키워드 분류 불일치
- 출처 링크 누락 또는 중복 데이터 존재 가능성

### 12.3 MVP 처리 정책
- 동일 판례의 중복 항목은 정리한다.
- 키워드 체계를 표준화해 검색 품질을 높인다.
- 출처가 불명확한 데이터는 공개 응답에 직접 사용하지 않거나 검수 상태로 분리한다.

### 12.4 판례 데이터 확보 및 내부 적재 정책
- 외부 공개 API가 불명확한 경우를 기본 전제로 두고, 서비스는 내부 DB 적재형 구조를 우선 채택한다.
- 초기 데이터 범위는 경찰 시험 준비에 직접적으로 필요한 형법, 형사소송법, 경찰학 주요 판례로 제한한다.
- 판례 데이터는 원문 텍스트와 메타데이터를 분리해 저장한다.
- 메타데이터 최소 항목은 사건번호, 판례명, 법원명, 선고일, 과목, 핵심 쟁점, 판결 요지, 시험 포인트, 출처 링크를 포함한다.
- 원문은 검색과 요약에 활용할 수 있도록 내부 DB에 저장하고, 검색용 청크와 키워드를 별도 인덱스로 관리한다.
- 판례 번호 입력은 내부 DB 사건번호 exact match를 우선 사용하고, 미등록 시 키워드 검색 또는 검수 대기 상태를 안내한다.
- OCR로 추출된 문제 지문 텍스트는 원문 확보 수단이 아니라 내부 판례 검색을 위한 입력 보조 데이터로 사용한다.
- LLM은 내부 DB에서 조회된 원문 또는 관련 청크를 기반으로만 요약과 질의응답을 생성한다.
- 검수되지 않은 요약 데이터는 공개 응답의 기준 데이터로 사용하지 않는다.

## 13. 비용 모델 (요약 중심 운영 기준)

### 13.1 비용 구조 요약
- 본 서비스의 핵심 비용 변수는 사용자 질의 수보다 월 신규/갱신 판례 처리 건수다.
- 사용자 조회 단계에서는 미리 생성된 요약을 반환하므로 실시간 LLM 호출 비중이 낮다.
- 비용은 고정비와 변동비로 분리 관리한다.

고정비 항목
- Apple Developer Program: 연 99 USD
- 서버/DB/스토리지/로그: 운영 규모에 따라 월 고정 발생

변동비 항목
- LLM 요약 생성 비용: 월 처리 판례 건수에 비례
- 임베딩 생성 비용: 신규/변경 청크 수에 비례
- App Store 수수료: 매출 발생 시 15~30%

### 13.2 월 LLM 비용 계산식
월 LLM 비용은 다음 식으로 계산한다.

월 LLM 비용 = N x ((Cin x Pin) + (Cout x Pout))

- N: 월 처리 판례 건수
- Cin: 판례 1건당 입력 토큰
- Cout: 판례 1건당 출력 토큰
- Pin: 입력 토큰 단가
- Pout: 출력 토큰 단가

### 13.3 운영 가정(기준값)
- 판례 1건당 평균 입력 토큰: 6000
- 판례 1건당 평균 출력 토큰: 700
- 월 처리 단위: 300건, 1000건, 3000건
- 인프라 고정비(초기 기준): 월 150000원 ~ 400000원

### 13.4 월 비용 시뮬레이션 (3안)
아래 표는 모델 단가가 서로 다른 상황을 가정한 예시다.

| 구분 | 월 처리 판례 | LLM 비용(저비용 모델) | LLM 비용(고성능 모델) | 인프라(예시) | Apple 계정 월 환산 | 총비용 범위(예시) |
|---|---:|---:|---:|---:|---:|---:|
| 보수 | 300건 | 약 3000원 ~ 30000원 | 약 30000원 ~ 180000원 | 150000원 | 약 11000원 | 약 164000원 ~ 341000원 |
| 기준 | 1000건 | 약 10000원 ~ 100000원 | 약 100000원 ~ 600000원 | 250000원 | 약 11000원 | 약 271000원 ~ 861000원 |
| 확장 | 3000건 | 약 30000원 ~ 300000원 | 약 300000원 ~ 1800000원 | 400000원 | 약 11000원 | 약 441000원 ~ 2211000원 |

주의
- 위 값은 단가 예시를 적용한 운영 계획용 추정치다.
- 실제 청구 금액은 선택 모델, 프롬프트 길이, 재시도율, 환율에 따라 변동된다.

### 13.5 비용 최적화 원칙
- 요약 재생성 조건을 엄격히 두고 원문 변경이 없는 데이터는 재처리하지 않는다.
- 초안 생성(저비용 모델)과 고품질 재요약(고성능 모델)을 분리 운영한다.
- 템플릿 기반 출력 형식을 고정해 출력 토큰을 통제한다.
- 실패 재시도 횟수 상한을 두고 비정상 입력은 검수 큐로 전환한다.

## 14. 요약 배치 파이프라인 운영안

### 14.1 처리 단계
1. 수집 큐 적재
- 신규/변경 판례만 ingestion_queue에 등록한다.

2. 전처리 및 정합성 점검
- 사건번호, 법원명, 선고일, 과목 필드 정규화
- 중복/누락 데이터 분리

3. 청크 생성 및 임베딩
- 원문을 고정 기준으로 분할 후 case_chunks에 저장
- 신규 청크만 임베딩 생성

4. LLM 요약 생성
- 표준 템플릿으로 쟁점/결론/시험 포인트 생성
- 상태값 summarized 반영

5. 검수
- 자동 규칙 검수(길이/누락/금칙어)
- 샘플 수동 검수 후 reviewed 반영

6. 게시
- reviewed 통과 건만 published 승격
- 서비스 조회는 published 데이터만 사용

### 14.2 권장 배치 스케줄
- 일간(매일): 신규/변경 판례 배치 실행, 실패건 재처리
- 주간(매주): 품질 지표 리포트, 상위 실패 원인 보정
- 월간(매월): 프롬프트/템플릿 개선, 비용 대비 품질 평가

### 14.3 운영 KPI
- 판례 1건당 평균 요약 비용
- 배치 성공률(성공 건수 / 시도 건수)
- 요약 검수 통과율
- published 전환 리드타임
- 재요약 비율(불필요 재처리 감시 지표)

### 14.4 장애 대응 기준
- LLM 호출 실패율 급증: 배치 중단 후 재시도 상한 축소
- 토큰 비용 급증: 출력 길이 정책 즉시 하향
- 품질 저하: 고성능 재요약 비율 임시 확대 후 원인 분석
